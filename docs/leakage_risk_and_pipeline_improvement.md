# 数据泄漏风险与 Pipeline 改进说明

## 1. 可能被严格评审质疑的点
当前实验流程中，以下步骤在全量数据上完成后再切分训练/测试集：
- `TotalCharges` 缺失值使用全量中位数填补
- `high_charge_low_tenure` 使用全量中位数阈值构造

这类做法会引入轻度“数据分布先验泄漏”风险，尽管未直接使用标签信息。

## 2. 为什么当前风险相对有限
- 没有使用 `Churn` 标签参与填补或阈值构造
- 使用的是非监督统计量（中位数）
- 任务定位为课程/实验流程，重点在方法演示与业务解释

因此该风险更多是“工程规范性问题”，而非严重目标泄漏。

## 3. 生产级改进方案
建议升级为严格训练期拟合、测试期应用的 Pipeline 流程：
1. 先执行 `train_test_split`（stratify 目标变量）
2. 仅在训练集上 `fit`：
   - 缺失值填补器（Imputer）
   - 类别编码器（One-Hot Encoder）
   - 特征阈值（如 `high_charge_low_tenure` 的中位数）
3. 将训练集拟合得到的参数统一应用到测试集
4. 使用 `sklearn Pipeline + ColumnTransformer` 串联全流程，避免手工步骤不一致

## 4. 未来改进方向
- 将预处理、特征工程、模型训练统一封装到单个可复现 Pipeline
- 增加时间切分或滚动验证，验证策略在时间维度上的稳健性
- 引入模型漂移与数据质量监控（缺失率、分布漂移、阈值稳定性）
- 建立阈值策略自动回测机制，结合成本敏感分析定期更新运营策略
